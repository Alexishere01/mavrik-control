# MAVRIK VTOL Simulation - Claude Code Onboarding

Welcome! This document provides the critical context, architecture, and commands you need to work within the MAVRIK VTOL Simulation project bridged to ArduPilot SITL.

## 1. Project Overview & Architecture
This project runs a high-fidelity 3D physics simulation (`mavrik.exe`) connected to an ArduPilot SITL (Software In The Loop) instance via a custom Python bridging script.
*   **Physics Engine:** `mavrik.exe` simulates a Quad-X VTOL with two forward tilt-rotors (Bicopter/Vectored Yaw style).
*   **Bridge:** `mavrik_ardupilot_bridge.py` routes MAVLink JSON data back and forth between the physics engine and ArduPilot on port `14552`.
*   **Web Viewer:** A Python HTTP server (`mavrik_web_viewer.py`) + WebSockets serving a Three.js 3D visualization on `http://localhost:8000`.

## 2. Execution Commands
Always use these shell scripts to manage the environment. Do not try to run the python scripts or `mavrik.exe` manually.

*   **Start the Simulation:** `./start_terminals.sh`
    *   This script spins up `sim_vehicle.py`, `mavrik.exe`, the bridge, and the web viewer in the background.
*   **Clean Up / Stop:** `./kill_zombies.sh`
    *   ALWAYS run this before restarting the simulation. It aggressively kills all orphaned Python and SITL processes to prevent port binding errors.

## 3. Critical Configuration Knowledge
We use ArduPlane (QuadPlane) to support differential tilt for yaw. If you ever modify `mavrik.parm`, you **MUST** strictly adhere to the following rules:

*   **Firmware:** ArduPlane (`sim_vehicle.py -v ArduPlane -f quadplane`).
*   **Vectored Yaw:** Uses `Q_TILT_TYPE 2`.
*   **Tilt Mask:** `Q_TILT_MASK 5` (Only Front Right and Front Left motors tilt). Do NOT use `15`, as the rear motors are fixed.
*   **Servo Mapping Constraints (CRITICAL):**
    *   The MAVRIK bridge maps ArduPilot's PWM output for the tilt servos (CH8/CH9) using an inverted formula: `1.0 - pwm_to_normalized(pwm_us)`.
    *   `1000` PWM = `1.0` (fully backward — not used in practice).
    *   `1500` PWM = hover → snap-band returns `VTOL_FLAP (0.99)`.
    *   `2000` PWM = `0.0` (FORWARD / Fixed Wing).
    *   Required settings:
        *   `SERVO8_TRIM 1500` and `SERVO9_TRIM 1500` — **CRITICAL**: centers the differential-yaw authority. With TRIM=1000, `Q_TILT_TYPE 2` has no room to tilt backward for yaw corrections, causing one motor to saturate toward 2000 (full FW tilt) during large yaw errors → uncontrolled forward transition → altitude OOB crash.
        *   `SERVO8_MIN 1000` and `SERVO8_MAX 2000`
        *   `SERVO9_MIN 1000` and `SERVO9_MAX 2000`
        *   `SERVO8_REVERSED 0` and `SERVO9_REVERSED 0`
        *   `Q_TILT_YAW_ANGLE 0`
*   **Mode switching:** `FLTMODE_CH 0` — disables RC-based mode switching. Without this, AP reads ch8=0 (no hardware RC) and immediately switches away from QHOVER after arm.
*   **Initial mode:** `INITIAL_MODE 18` (QHOVER). Ensures AP starts in the correct VTOL mode before the web viewer fires SET_MODE.

## 4. Logging & Debugging
If the drone behaves erratically, check the following files:
*   `bridge_diag.csv`: High-frequency telemetry logged directly by the Python bridge. Columns: `t, sim_t, alt_ft, roll, pitch, yaw, u, v, w, p, q, r, ch1–ch4, ail, elev, rud, thrFR, thrFL, thrAR, thrAL, flapsR, flapsL, armed, ap_ch1–ch4, ap_ch8, ap_ch9, ap_armed, motor_active`. Excellent for checking raw MAVRIK state and the exact `c[0]-c[8]` control outputs.
*   `mav.tlog`: Binary MAVLink log generated by the SITL. Use `pymavlink` to parse `SERVO_OUTPUT_RAW` and `ATTITUDE` to verify what ArduPilot was commanding internally.

## 6. Bridge Control Conventions (Hard-Won)

These took multiple diagnostic runs to establish. Do not change without re-verifying against `bridge_diag.csv`.

### Flap / Tilt Mapping
MAVRIK's flap convention is **inverted** relative to ArduPilot's PWM convention:

| ArduPilot PWM | MAVRIK flap value | Meaning |
|---|---|---|
| 1000 | 0.99 (VTOL_FLAP) | Motors pointing UP — VTOL hover |
| 2000 | 0.0 | Motors pointing FORWARD — fixed-wing |

The bridge therefore uses `1.0 - pwm_to_normalized(pwm_us)` for tilt channels CH8/CH9.

**Neutral snap band**: `[1440, 1560]` → `VTOL_FLAP (0.99)`. With `SERVO8_TRIM=1500`, AP outputs exactly 1500 at hover. The band absorbs small differential-yaw offsets (±40 PWM) without letting them partially de-tilt the rotors. Do NOT narrow this band. Do NOT use `SERVO8_TRIM=1000` — that removes backward yaw authority and causes the tilt controller to saturate toward 2000 (full FW) during large yaw errors, crashing the MAVRIK atmosphere model.

### Motor Channel Mapping (verified empirically)
```
CH1 → thrFR  (Front Right)
CH2 → thrAL  (Aft Left)
CH3 → thrFL  (Front Left)
CH4 → thrAR  (Aft Right)
```
MAVRIK control packet order: `(aileron, elevator, rudder, thrFR, thrFL, thrAR, thrAL, flapsRight, flapsLeft)`

Note the packet has `thrFR, thrFL, thrAR, thrAL` — ArduPilot CH2 is thrAL (not thrAR). Swapping these causes cross-coupled roll/pitch.

### Three-Phase Bridge Architecture (Working as of 2026-05-10)

The bridge runs at ~2 kHz (non-blocking sockets + `time.sleep(0.0005)`). The pre-arm → handover → flight sequence is:

#### Phase 1 — No MAVRIK yet (synthetic state)
- MAVRIK hasn't connected. Bridge sends synthetic JSON state (frozen level, altitude = 3000 ft) to ArduPilot so SITL doesn't time out and crash.
- **pid_vtol.py owns port 5006** and sends cached trim (0.125 symmetric, all 4 motors) at 2 kHz before MAVRIK connects. MAVRIK's first `recvfrom(5006)` is always satisfied immediately.
- Bridge does NOT send anything to MAVRIK in this phase. The old `PRE_ARM_FREEFALL` send has been removed — at 2 kHz loop rate it raced equally with pid_vtol's 0.125 trim sends, causing alternating 0/0.125 commands → MAVRIK pitch diverged 5° in 1.4s → crash.

#### Phase 2 — MAVRIK connected, pre-arm
- MAVRIK state arrives on port 5009. Bridge validates 3+ consecutive valid packets, sets `_mavrik_connected=True`.
- Bridge sends `make_level_proxy(state)` to ArduPilot — this forces level quaternion (`e0=1, ex=ey=ez=0`), zeroes all body velocities and rates, but **passes through real `zf` (altitude)**.
    - Passing real `zf` is critical: old code froze `zf = -3000.0` during pre-arm, then jumped to real `zf` (e.g. -3005.7) at handover → 1.74 m position step in one 5ms frame → EKF computed 348 m/s implied velocity → SITL crash.
- **pid_vtol.py still owns port 5006** and controls MAVRIK with its full cascaded PID (altitude, pitch, roll, yaw loops). It holds MAVRIK level and at 3000 ft until the web viewer fires the ARM command.
- Bridge does NOT send to port 5006 in this phase.

#### Handover gate (Phase 2 → 3)
Fires when:
1. AP's PWM frame shows any motor > 1000 µs (ArduPilot has armed internally)
2. `|roll| < 15°` and `|pitch| < 15°` (attitude is level enough for AP to take over)

On handover:
- Bridge sends UDP byte `\x01` to port 5015 → pid_vtol.py stops its control loop and releases port 5006.
- `_handover_grace_until = now + 0.15` (150ms grace window).

#### Grace period (150ms)
- Bridge sends `PRE_ARM_HOVER = (0.0, 0.0, 0.0, 0.394, 0.394, 0.410, 0.410, VTOL_FLAP, VTOL_FLAP)` to MAVRIK at 2 kHz for 150ms.
- This covers the ~20ms pid_vtol wind-down. pid_vtol also sends hover trim during its last ticks — both sides agree on hover trim, no physics conflict.

#### Phase 3 — ArduPilot in control
- Grace period over. Bridge forwards AP's PWM commands directly to MAVRIK.
- Bridge sends real MAVRIK state (no level proxy) to ArduPilot.
- RC override socket (port 5012) is bound lazily — rate-limited to ~2 Hz on bind retry to avoid leaking 2000 OS file descriptors/second during the brief window when pid_vtol still holds port 5012.

### Pre-Arm Behavior
```python
PRE_ARM_HOVER = (0.0, 0.0, 0.0, 0.394, 0.394, 0.410, 0.410, VTOL_FLAP, VTOL_FLAP)
```
These throttle values match `Team1_VTOL.json` initial trim exactly. **Do not replace this with a PD attitude controller** — at 2 kHz MAVRIK update rate, even KP=0.6 causes violent limit cycles and the drone arms inverted.

The bridge does NOT send `PRE_ARM_FREEFALL` (all zeros) before MAVRIK connects. `pid_vtol.py` exclusively owns port 5006 during pre-arm and handles MAVRIK stabilization. Do not re-add the freefall send.

### Auto-Arm Sequence (web viewer)
`mavrik_web_viewer.py` runs `auto_arm_loop()` which:
1. Sends `SET_MODE(18)` (QHOVER) 5 times × 0.1s = 0.5s.
2. Waits for stable attitude: `|roll| < 10°` and `|pitch| < 10°` for **2.0 continuous seconds**.
    - Do NOT reduce this below 2.0s — MAVRIK needs time to settle after connecting and `pid_vtol.py` needs multiple ramp seconds to bring attitude within bounds. At 0.5s the ARM fired before stability was achieved, causing MAVRIK crash ~1.4s later.
3. Sends MAVLink ARM command to ArduPilot.
4. Returns — does NOT send RC_CHANNELS_OVERRIDE after arm. ArduPilot's QHOVER manages altitude autonomously once armed.

## 7. PID Tuning — Current Values and History

Current `mavrik.parm` PID values (as of 2026-05-07):

```
Q_A_RAT_PIT_P  0.15    Q_A_RAT_RLL_P  0.15    Q_A_RAT_YAW_P  0.18
Q_A_RAT_PIT_I  0.10    Q_A_RAT_RLL_I  0.10    Q_A_RAT_YAW_I  0.015
Q_A_RAT_PIT_D  0.003   Q_A_RAT_RLL_D  0.003   Q_A_RAT_YAW_D  0.002
Q_A_RAT_PIT_IMAX 0.30  Q_A_RAT_RLL_IMAX 0.30  Q_A_RAT_YAW_IMAX 0.30

Q_A_ANG_PIT_P  3.0     Q_A_ANG_RLL_P  3.0     Q_A_ANG_YAW_P  2.0

MOT_THST_HOVER 0.40
```

**Why these values** (abbreviated tuning history):

- `Q_A_ANG_*_P`: Started at 2.0 (too weak — only 6% of time near-level). Raised to 4.5 → visible ~0.4s roll oscillations. Settled at **3.0**.
- `Q_A_RAT_*_P`: Started at 0.08–0.12. Raised with angle gains. Settled at **0.15** (pitch/roll) and **0.18** (yaw).
- `IMAX 0.30`: Added after observing integrator windup during large attitude excursions causing the drone to saturate motors in one direction indefinitely.
- `MOT_THST_HOVER 0.40`: Matches MAVRIK trim (avg of 0.394/0.410 front/aft). Too low (0.35) caused uncontrolled climbs to 4000+ ft from excess thrust margin.

**Loiter / position limits** (set conservatively to prevent tumble from over-aggressive position demands):
```
LOIT_ACC_MAX 75       # cm/s² → ~4.4° pitch demand max
LOIT_BRK_ACCEL 50
LOIT_SPEED 200
PILOT_SPEED_UP 150    # cm/s
PILOT_SPEED_DN 75
PILOT_ACCEL_Z 100
```

## 8. Diagnostic Workflow

When the drone behaves erratically, follow this order:

1. **Check `bridge_diag.csv`** — Verify:
   - `flapsR` and `flapsL` (c[7], c[8]) ≈ 0.99 throughout VTOL. If ~0.5, the tilt neutral snap band missed AP's actual output.
   - Motor values `thrFR/FL/AR/AL` symmetric around ~0.40 at hover. Large asymmetry = motor mapping error or attitude error.
   - `roll`/`pitch` at the moment `armed` transitions 0→1. Arming at >30° is a pre-arm behavior bug.
   - If pitch diverges in pre-arm phase: look for `PRE_ARM_FREEFALL` being sent — the bridge must NOT send anything to port 5006 before arm. pid_vtol owns it exclusively.

2. **Parse `mav.tlog`** with pymavlink to get `SERVO_OUTPUT_RAW` — compare CH8/CH9 raw PWM to what `_tilt()` would output. If AP is sending something unexpected, the `.parm` file may not have loaded (check for `SERVO8_TRIM` in the log).

3. **Near-level percentage**: Count rows where `abs(roll)<5 and abs(pitch)<5`, divide by total. Below ~40% in QHOVER means PID gains are insufficient or attitude at arm was too large.

4. **Connection drops after handover**: If QGC loses connection immediately after handover prints, the most likely cause is an EKF crash from a position step. Check `zf` in `bridge_diag.csv` — if it jumps >0.5 ft in a single row at the handover timestamp, `make_level_proxy` is freezing altitude (regression). Real `zf` must pass through during pre-arm.

## 9. Working State Summary (2026-05-10)

First successful sustained ArduPilot-controlled VTOL flight achieved. The full sequence works:

1. `./start_terminals.sh` launches all processes.
2. MAVRIK connects, bridge Phase 2 begins. pid_vtol stabilizes drone via cascaded PID.
3. Web viewer sends SET_MODE(QHOVER) × 5, then waits 2.0s stable attitude.
4. Web viewer sends ARM. ArduPilot arms. Bridge detects motor PWM > 1000.
5. Handover gate passes (|R|<15°, |P|<15°). Bridge stops pid_vtol, enters 150ms grace.
6. Grace ends. Bridge forwards AP commands to MAVRIK. Real state to ArduPilot.
7. QGC stays connected. QHOVER holds altitude. Drone flies stably.

Key bugs fixed to get here:
- Removed `PRE_ARM_FREEFALL` from bridge's 2kHz loop (raced with pid_vtol's 0.125 trim → MAVRIK crash).
- `make_level_proxy` passes through real `zf` (frozen altitude → EKF velocity spike at handover → SITL crash).
- RC socket bind rate-limited to ~2 Hz (2kHz OS socket creation → FD leak).
- Stable timer set to 2.0s (0.5s was too short, ARM fired before attitude settled).
- RC override loop removed after arm (AP QHOVER manages altitude autonomously).
